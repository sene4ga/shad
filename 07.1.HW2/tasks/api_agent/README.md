## Simple Agent API

`rest` `api` `http` `sse` `mcp` `llm` `agent`

В этой домашке вам предлагается сделать HTTP API, под капотом которого живет простой агент. Агент должен уметь работать с внешней LLM, вызывать тулы из подключенных MCP-серверов и сохранять историю чатов.

Цель задания - потренироваться писать асинхронный код, хранить необходимые данные в БД, разобраться с [Chat Completions API](https://developers.openai.com/api/reference/resources/chat) и [MCP](https://modelcontextprotocol.io/docs/getting-started/intro), а заодно спроектировать внятный API под пользовательские сценарии.

### Условие

Нужно реализовать API, которое позволяет пользователю:

1. добавлять свои конфиги LLM;
2. добавлять внешние Streamable HTTP MCP;
3. подключать и отключать MCP в конкретном чате;
4. добавлять сообщения в чат;
5. читать историю чата;
6. не терять настройки и историю чатов (данные не должны удаляться);
7. подключать к любому из чатов заранее реализованный MCP с ровно двумя арифметическими операциями: умножение аргумента на два и деление одного числа на другое.

Как именно вы это реализуете - не важно. Важна функциональность и пользовательские сценарии, а не конкретный фреймворк, роутер, ORM, база данных или способ организации кода.

### Требования

1. API должно поддерживать аутентификацию через HTTP headers. Мы хотим, чтобы сервис не был полностью открыт во внешний интернет и чтобы данные разных пользователей были изолированы друг от друга.
2. Минимально допустимый вариант аутентификации:
   - сервер проверяет фиксированный секретный токен, переданный в одном из headers;
   - сервер получает идентификатор пользователя из отдельного header;
   - все данные должны храниться и читаться в разрезе этого пользователя.
3. Более продвинутый, но не обязательный вариант:
   - можно реализовать регистрацию или login;
   - можно выдавать JWT или другой bearer token;
   - можно кодировать пользователя внутри токена.
4. Для сдачи достаточно и простого варианта, если выполняются следующие условия:
   - каждый endpoint защищен проверкой auth headers;
   - запросы без корректной аутентификации отклоняются;
   - пользователь A не может увидеть чаты, сообщения, MCP-конфиги, LLM-конфиги и настройки пользователя B. При этом кто из пользователей к нам пришел говорит сам пользователь :)
5. Идентификация пользователя должна происходить из headers, а не из тела запроса.
6. При добавлении конфига LLM пользователь предоставляет `BASE_URL`, `API_KEY` и `MODEL`.
7. Пользователь может добавить конфигурацию для подключения к MCP серверу:
   - пользовательский конфиг MCP должен состоять только из `URL`, `TOKEN` и `NAME`;
   - transport для таких пользовательских MCP: только [Streamable HTTP](https://modelcontextprotocol.io/specification/2025-03-26/basic/transports#streamable-http);
   - авторизация для таких пользовательских MCP: достаточно передавать токен в заголовке `Authorization: Bearer <token>`.
8. Если в рамках агентской итерации один и тот же tool был вызван более 2 раз подряд, агентский цикл должен быть прерван и клиент должен получить сообщение об ошибке.
9. Если агентская итерация заняла более 10 completions, агентский цикл должен быть прерван и клиент должен получить сообщение об ошибке.
10. Изменение списка MCP, подключенных к чату, может происходить только между итерациями агентского цикла. Со следующей итерации список тулов, доступных LLM, должен обновиться.

### Ограничения и упрощения

Чтобы задание оставалось выполнимым и проверяемым, в рамках этой домашки достаточно поддержать только OpenAI-совместимый endpoint `POST /v1/chat/completions`. Более свежий и рекомендуемый везде протокол Responses API пока что за рамками этого задания.

Для простоты также **не требуем** поддержку всех возможных вариантов MCP. Достаточно поддержки Streamable HTTP MCP.

При этом встроенный MCP сервер с арифметическими tools, который доступен всем пользователям по умолчанию, можно реализовать как угодно - в том числе через `stdio`, если вам так удобнее.

Важно: в этой задаче асинхронность особенно важна для LLM-запросов, SSE-streaming и вызовов tools, потому что именно там возникает заметная сетевая задержка. Способ хранения состояния можно выбирать свободно: подойдет `sqlite`, любая другая СУБД или даже просто файл на сервере, если требования по функциональности соблюдены. Запись-чтение может быть синхронным, ничего страшного.

Любые запросы пользователя, которые приходят в процессе работы агентской итерации, могут игнорироваться с поясняющим сообщением, могут быть поставлены в очередь, а могут быть обработаны каким-то иным разумным способом. Проще говоря, преподаватели не будут специально ломать ваше решение именно такой гонкой. Другими способами - да, этим - нет.

### Как обеспечить бесперебойную работу сервиса?
Можете воспользоваться [Yandex Cloud](https://cloud.yandex.ru/), любым другим облачным провайдером или своим собственным сервером, если он у вас есть.

Для Yandex Cloud вам будет предоставлен грант. С помощью него вы сможете получить доступ к LLM, доступ на виртуалку вам также будет организован. Для получения доступа на виртуалку и граната, необходимо заполнить [форму](https://forms.yandex.ru/surveys/13844735.cd98f5d71cda9e18f27e206c42ddf47908fa82f3/). После заполнения формы на указанную там почту придет письмо с данными для доступа на виртуалку. Как письмо с именем пользователя и паролем для доступа на виртуалку, так и грант на облаке выдаются в полу-ручном режиме, поэтому приходят не мгновенно (возможна задержка до нескольких дней). Чем раньше заполните форму, тем лучше. Сервер доступен публично по домену [ai360.velkerr.ru](ai360.velkerr.ru). На одной виртуальной машине будут одновременно задеплоены все решения, поэтому придется делиться портами. Проверить, занят ли порт, можно через `lsof -i :<номер порта>`. Чтобы избежать коллизий, предлагаю взять порт 40000 + номер из логина на виртуалке. Например, студент с логином `ai360p2026s110` занимает порт 40110. В любом случае, домен и порт нужно указать в описании МРа. 

К сожалению, Pycharm Community не поддерживает подключения по SSH.
Для разработки на удаленном сервере можно использовать Pycharm Pro, VS Code или разрабатываться локально и подтягивать изменения на сервере через git. В случае доставки через гит, рекомендую добавить в свой репозиторий на [https://gitlab.manytask.org/](https://gitlab.manytask.org/) read only [deploy key](https://docs.gitlab.com/user/project/deploy_keys/) и подтягивать изменения на виртуалку через `git pull`.

На сервере нужно настроить виртуальное окружение (напишите `pyproject.toml` для своего сервиса). Далее нужно запустить сервис. Возможны разные варианты как это сделать, самый простой - открыть `tmux` и запустить что-то вроде `PORT=40110 python <имя файла с приложением>`. Более интересный - задеплоить приложение внутри докер контейнера. К слову, тогда и настраивать среду не понадобится.

### Отправка задания

1. Выполняйте задание в отдельной ветке.
2. Обеспечьте бесперебойную работу вашего API, задеплоив сервис на яндекс облаке (или в другом облачном сервисе).
3. Отправьте Merge request к себе в репозиторий, поставьте ему в GitLab label `api_agent`.
**Важно:** не вмердживайте MR сами. Ждите проверки.
4. В описании MR укажите:
   - базовый URL вашего сервиса;
   - как именно проверяющему аутентифицироваться;
   - если сервис недоступен без специальных headers, приведите пример нужных headers.
5. Сервис должен предоставлять описание API в виде OpenAPI схемы (swagger.json), [ссылка](https://www.compilenrun.com/docs/framework/fastapi/fastapi-documentation/fastapi-openapi-schema/), которая рендерится по адресу /docs, чтобы преподаватели могли понять как пользоваться сервисом.

После этого к вам придет проверяющий и проверит, что API работает, а также проглядит код.

### Замечания

1. В этом задании нет никаких автоматических тестов: мы проверяем руками, как работает ваш API. Однако вам нужно приложить код через MR, мы кратко оценим его без полноценного code review.
2. Приватные токены любого вида в репозиторий коммитят только редиски 😛. Используйте переменные окружения (чаще всего, файл `.env`, который есть на сервере, но не закоммичен в git, см `.gitignore`) или любой другой способ передавать секреты. Авторизационный токен, если используется "простая" авторизация, также не должен быть захардкожен в коде сервиса, но при этом лежать в тексте MR.
3. Если в описании MR нет URL сервиса или не описано, как пройти аутентификацию, мы не сможем проверить решение.
4. Грант на AI Studio нужен для вас, для тестирования в процессе разработки. Не нужно предоставлять пользователю LLM с вашего гранта в качестве запасного варианта. Пользователь сам приносит конфиг LLM. Также, чтобы максимизировать вероятность того, что гранта хватит, старайтесь использовать модельку поменьше, например, `yandexgpt-lite` или gpt-oss20b, цены [тут](https://aistudio.yandex.ru/docs/ru/ai-studio/pricing.html).
